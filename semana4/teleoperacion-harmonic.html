Cambios en el plugin de la persona

Durante la migración del proyecto a Gazebo Harmonic, se detectó que el plugin de la persona implementado originalmente para Gazebo Classic no era compatible con la nueva arquitectura.

Diferencias en los includes

En Gazebo Classic se utilizaban las siguientes librerías: 

#include <gazebo/gazebo.hh>
#include <gazebo/physics/physics.hh>
#include <gazebo/common/common.hh>
#include <ignition/math/Quaterniond.hh>

En Gazebo Harmonic, estas dependencias cambian completamente hacia el ecosistema gz-sim: 

#include <gz/sim/System.hh>
#include <gz/sim/Model.hh>
#include <gz/sim/Util.hh>
#include <gz/math/Quaterniond.hh>

Cambio en la estructura del plugin

El cambio más importante es que en Gazebo Classic se utilizaba un ModelPlugin: 

class Person : public ModelPlugin

En Gazebo Harmonic esto deja de existir. Ahora los plugins se implementan como sistemas ECS: 

class PersonSystem :
  public gz::sim::System,
  public gz::sim::ISystemConfigure,
  public gz::sim::ISystemPreUpdate

Esto implica que la lógica ya no se basa en callbacks de modelo, sino en fases del ciclo de simulación.

Registro del plugin

En Gazebo Classic el plugin se registraba de forma estática con: 

GZ_REGISTER_MODEL_PLUGIN(Person)

En Gazebo Harmonic este sistema desaparece. En su lugar, el plugin se registra mediante el sistema de factories de gz-sim. Ya no existe un macro equivalente directo.

Actualización del modelo

En Classic se modificaba directamente la pose del modelo: 

this->model->SetWorldPose(pose);

En Harmonic, el acceso al modelo se realiza a través del EntityComponentManager, modificando componentes en lugar de manipular directamente el objeto.

Conclusión

La migración a Gazebo Harmonic supuso una reestructuración importante del plugin original. Aunque fue necesario modificar la arquitectura y las dependencias, el sistema final sigue permitiendo el movimiento de la persona dentro de la simulación.

En particular, el nuevo enfoque mantiene la capacidad de mover al agente tanto de forma autónoma como mediante control externo, asegurando la continuidad funcional del sistema tras la migración.
