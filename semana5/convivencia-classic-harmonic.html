Convivencia entre Gazebo Classic y Gazebo Harmonic

Esta semana se consiguió resolver el problema de compatibilidad entre los plugins de Gazebo Classic y Gazebo Harmonic, permitiendo su convivencia dentro del mismo workspace sin errores de compilación.

Problema inicial

Ambos entornos utilizan arquitecturas completamente diferentes, lo que generaba conflictos al intentar compilar ambos plugins en el mismo CMake:

  • Gazebo Classic → API basada en gazebo::physics y ModelPlugin
  • Gazebo Harmonic → arquitectura basada en gz-sim (ECS)

Esto provocaba errores de compilación y linkado debido a dependencias incompatibles.

Solución adoptada

La solución consistió en separar completamente la compilación de ambos plugins dentro del CMake, evitando que compartieran el mismo flujo de dependencias.

Para ello se aplicaron dos ideas principales:

  • Separación de targets en CMake (Classic vs Harmonic).
  • Aislamiento de dependencias por plugin.

Uso de OBJECT library en Classic

Para el plugin de Gazebo Classic se utilizó una OBJECT library, lo que permite compilar el código fuente sin mezclarlo directamente con el target final: 

add_library(personplugin_obj OBJECT amazon_hospital/src/person.cpp)

add_library(personplugin STATIC $)

ament_target_dependencies(personplugin "rclcpp" "pluginlib" "gazebo")

Este enfoque evitó conflictos con dependencias internas del sistema Classic.

Plugin Harmonic independiente

El plugin de Gazebo Harmonic se mantuvo completamente separado, compilando directamente contra las librerías de gz-sim: 

add_library(personplugin_harmonic follow_person_harmonic/src/person_harmonic.cpp)

target_link_libraries(personplugin_harmonic
  gz-sim8::gz-sim8
  gz-plugin2::gz-plugin2
  gz-math7::gz-math7
  Threads::Threads)

Importancia del uso de subdirectorios en CMake

Además de separar los targets, fue clave estructurar la compilación mediante subdirectorios en el CMake, de forma que cada plugin se compila desde su propia ruta y contexto.

Esto permite que CMake trate cada plugin como una unidad independiente de compilación, evitando mezclar includes o dependencias entre Classic y Harmonic.

Ejemplo de separación por subdirectorios en CMake

# Plugin Classic (carpeta amazon_hospital/)
add_library(personplugin_obj OBJECT amazon_hospital/src/person.cpp)
add_library(personplugin STATIC $)

# Plugin Harmonic (carpeta follow_person_harmonic/)
add_library(personplugin_harmonic follow_person_harmonic/src/person_harmonic.cpp)

Gracias a esta separación, cada plugin se compila dentro de su propio contexto, evitando conflictos de includes y resolviendo correctamente sus dependencias.

Resultado

Esta combinación de separación de targets y uso de subdirectorios en el CMake permitió compilar ambos plugins en el mismo proyecto sin errores.

Además, se pudo mantener la versión Classic como referencia mientras se desarrollaba la versión Harmonic en paralelo.

Conclusión

La solución adoptada permitió una migración progresiva sin romper el sistema existente, garantizando compatibilidad entre ambos entornos de Gazebo.
